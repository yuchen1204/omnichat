三种重构方案

从当前 6500+ 行到极简 Agent 工具

方案 A：最小可行移植 (推荐)

直接对标 Claude Code AgentTool，只移植核心执行循环

方案 A 架构
新建文件 (3 个)
AgentTool.kt — 工具定义 + 执行入口
AgentRunner.kt — SubAgent 隔离执行循环
AgentContext.kt — Agent 上下文 (精简版)
删除文件 (12+ 个)
MessageBus.kt · TaskManager.kt · Scratchpad.kt
AgentExecutionLoops.kt · OrchestratorTools.kt
AgentLifecycle.kt · AgentRegistry.kt · AgentDefinition.kt
AgentTask.kt · ModelSelector.kt · ToolOrchestrator.kt
AgentProgressSummarizer.kt · WorkspaceModels.kt (大部分)
保留/修改
TeamManager.kt — 精简为薄门面 (~100 行)
TeammateContext.kt — 保留身份传播
WorkspaceModels.kt — 只保留 TeamState 等 UI 类型
Agent 工具 Schema
description: String
prompt: String
model?: String
subagent_type?: String
预估代码量
新增: ~400 行
删除: ~5500 行
保留: ~1000 行
净减: ~5100 行

优点

  • 代码量减少 ~80%
  • 直接对标 Claude Code，可验证
  • SubAgent 上下文完全隔离
  • 无 MessageBus/TaskManager 复杂性
  • 后续加 B (异步) 不需要重构

缺点

  • SubAgent 之间不能直接通信
  • 没有 Claim 模式的自动任务分配
  • 需要重写 Orchestrator 系统 prompt

方案 B：渐进式重构

保留当前文件结构，逐步删除不需要的模块

优点

  • 每一步都可编译可测试
  • 风险更低，可以随时停下来

缺点

  • 中间态代码仍然复杂
  • 容易陷入"改了一半"的泥潭
  • 最终结果和方案 A 相同但耗时更长

方案 C：全新模块

在新 package 下从零实现，旧代码保留但不使用

优点

  • 完全干净的新代码，无历史包袱
  • 旧代码可作为参考
  • 可以并行开发，不影响现有功能

缺点

  • 需要处理 UI 层的适配
  • 旧代码不删会成为噪音
  • 两个系统并存增加认知负担
