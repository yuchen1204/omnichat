Claude Code vs OmniChat 当前架构对比

核心差异一目了然

Claude Code 模式 (目标)
极简工具 Schema
Agent(
  description: "3-5 词描述",
  prompt: "完整任务描述",
  subagent_type?: "Explore" | "Plan" | ...,
  model?: "sonnet" | "opus" | "haiku",
  run_in_background?: boolean
)
执行流程
1. Lead Agent 调用 Agent 工具
2. 选择 AgentDefinition (预设)
3. 克隆上下文，创建隔离环境
4. 启动 runAgent() AsyncGenerator
5. SubAgent 独立执行，流式返回消息
6. 收集最后一条 assistant 消息作为结果
7. 清理资源，返回结果给 Lead Agent
通信
SubAgent → 结果返回 Lead Agent
SubAgent 之间 不通信
OmniChat 当前模式 (要替换)
复杂工具 Schema
create_agents(
  agents: [{
    name, role, systemPrompt,
    modelConfigId, modelHint,
    dependsOn, spawnMode,
    taskMode, task, description
  }]
)
assign_task(target, task)
continue_conversation(target, msg)
peer_message(to, message)
执行流程
1. Orchestrator 调用 create_agents
2. 解析复杂 JSON (3 层 fallback)
3. 创建 AgentContext + AgentRunner
4. 启动 runTeammateLoop() 协程
5. Agent 进入 idle → execute → idle 循环
6. 通过 MessageBus 发送 ResultReport
7. Orchestrator 收到
通信
Orchestrator ↔ MessageBus ↔ SubAgent
SubAgent ↔ peer_message ↔ SubAgent

要删除的模块

✗

MessageBus

Channel-based 消息总线，256 容量收件箱，死 Agent 保护。Claude Code 模式不需要。

✗

TaskManager + Claim 模式

Room 持久化的任务队列，原子认领，依赖阻塞。过度设计。

✗

AgentExecutionLoops (1245 行)

idle → execute → idle 循环，waitForNextMessage 优先级，ResultReport 格式化。全部重写。

✗

OrchestratorTools 复杂路由

4 个工具的拦截路由，3 层 JSON fallback 解析。简化为单一 Agent 工具。

✗

Peer Messaging

SubAgent 间直接通信，广播模式。Claude Code 模式下 SubAgent 互相隔离。
